> **Estado:** _borrador_

You don’t “create KTX2 textures”, you **encode** existing images (or procedurally generated images) into a **KTX2 container** using a compressor/transcoder.

Here’s the practical pipeline plus the minimal theory you might want.

---

## 1) Practical: Convert PNG/JPG → KTX2 (recommended toolchain)

### Option A — Khronos `toktx` (official, best default)

Install **KTX-Software** (contains `toktx`).

Then:

```bash
# High-quality (UASTC) good for normal maps / crisp textures
toktx --uastc --zstd 18 output.ktx2 input.png

# Smaller (ETC1S) good for albedo/basecolor on web/mobile
toktx --bcmp --zstd 18 output.ktx2 input.png
```

Add mipmaps (almost always):

```bash
toktx --uastc --genmipmap --zstd 18 output.ktx2 input.png
```

Normal map hint (improves compression choices):

```bash
toktx --uastc --normal_map --genmipmap --zstd 18 normal.ktx2 normal.png
```

**When to pick what**

- **UASTC**: higher quality, larger files, best for normals/detail.
- **ETC1S**: smaller, slightly blurrier, best for color/albedo on web.

---

### Option B — Basis Universal encoder (`basisu`)

Also common in web pipelines:

```bash
basisu -ktx2 -uastc -mipmap input.png
# or smaller:
basisu -ktx2 -etc1s -mipmap input.png
```

## 2) “Math” (what’s actually happening)

### A) Block compression (core idea)

Most GPU texture compression is **block-based**:

- The image is split into fixed-size blocks (often **4×4** texels).
- Each block is approximated by a small set of parameters (endpoints + indices).
- This is a **lossy approximation** that minimizes error.

So “math-wise,” encoding is an optimization problem:

[
\min_{\theta}; |B - \hat{B}(\theta)| \quad \text{s.t.} \quad \theta \in \text{finite parameter set}
]

Where:

- (B) is an original block
- (\hat{B}(\theta)) is the reconstructed block
- (\theta) are the compressed block parameters

Encoders do heuristic search because the exact optimum is expensive.

---

### B) Rate–distortion trade-off

Encoders pick parameters balancing:

- **Rate** (bits / file size)
- **Distortion** (visual error)

Conceptually:

[
\arg\min_{\theta}; D(B,\hat{B}(\theta)) + \lambda R(\theta)
]

You don’t set (\lambda) directly; you choose **UASTC vs ETC1S**, quality flags, and zstd level.

---

### C) Mipmaps (why they matter)

Mipmaps are prefiltered downscaled levels that reduce aliasing and improve performance.

They’re generated by filtering:

[
I_{k+1}(x,y) = \sum_{i,j} w_{ij}, I_k(2x+i,2y+j)
]

In practice, tools do this for you (`--genmipmap`).

---

## 3) If you truly mean “create” from math (procedural textures)

You can procedurally generate an image (noise, SDFs, gradients, etc.) in code, write it to PNG, then encode to KTX2.

Typical flow:

1. Generate `texture.png` procedurally (Python/GLSL/P5.js).
2. `toktx ... texture.ktx2 texture.png`.


---

## 4) Quick rules for your R3F / Three.js world

- Use **ETC1S** for large color textures (faster download).
- Use **UASTC** for normals/roughness/metalness if artifacts matter.
- Always **mipmaps**.
- Keep power-of-two dimensions when possible (helps mip chain & compatibility).
