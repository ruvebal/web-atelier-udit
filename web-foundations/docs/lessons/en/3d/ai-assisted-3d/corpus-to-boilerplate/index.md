# Corpus to Boilerplate — Master Prompt File

**Author / Course Direction:** Prof. Rubén Vega Balbás, PhD `<ruben.vega@udit.es>`

> *"A corpus becomes a tool only when it consents to become structure."*

This resource is a **prompt-in-a-file**: a master instruction set you can paste into Cursor, Windsurf, Claude, or another coding agent to transform the entire `web-foundations/docs/lessons/en/3d/` corpus into a real project boilerplate.

The point is not to generate "some 3D app". The point is to generate a **disciplined, production-oriented scaffold** whose architecture is grounded in the lesson corpus you already built — with a concrete 2D interface layer (shadcn/ui) that lives both **over** and **inside** the 3D space.

---

## What this file is for

Use this file when you want an AI coding agent to:

- read the whole 3D lesson corpus as a specification
- extract the architectural decisions already encoded in the lessons
- create a runnable boilerplate with real files and directories
- preserve the course philosophy: performance, clarity, accessibility, and AI-assisted rigor
- scaffold a 2D UI system (shadcn/ui + Tailwind) layered on top of and within the 3D canvas

This is especially useful when starting:

- a studio-style portfolio scaffold
- a 3D editorial app shell
- a React + R3F starter for advanced teaching
- a baseline repo for future assignments or student cohorts

## 2D-over-3D architecture

The boilerplate must implement a clean separation of the 2D interface layer:

1. **DOM overlay** — shadcn/ui components rendered above the Canvas (navigation, panels, controls, dialogs). These use standard React + Tailwind and never import three.js.
2. **Inner HTML** — shadcn/ui components rendered inside 3D objects using Drei's `<Html>` component. These follow 3D transforms (position, rotation, scale) and are used for labels, tooltips, interactive panels attached to meshes.
3. **Canvas layer** — R3F scene graph: meshes, lights, shaders, post-processing. Pure three.js/R3F — no DOM imports.

The boundary is strict:
- `src/components/ui/` contains ONLY DOM components (shadcn/ui, Tailwind, no three imports)
- `src/components/canvas/` contains ONLY R3F components (may use Drei Html to render from ui/)
- `src/components/canvas/overlays/` contains components that use Drei Html to bridge DOM content into the 3D scene

## Foundation layer: SSR entry points

The boilerplate must ship with React Router v7 SSR streaming entry points:

1. **`entry.server.jsx`** — Streaming SSR with `renderToPipeableStream`. Detect bots via `isbot` (serve `onAllReady` for crawlers, `onShellReady` for real users). Set `streamTimeout`. Include `handleError` that silences aborted requests. Use `ServerRouter` from `react-router`.
2. **`entry.client.jsx`** — Client hydration via `ReactDOM.hydrateRoot` with `HydratedRouter`. Import i18n client config. Disable browser native scroll restoration (`history.scrollRestoration = 'manual'`) so React Router's `ScrollRestoration` handles it.

This is not optional. SSR entry points define the rendering contract. Without them, the scaffold degrades to a toy SPA.

## Foundation layer: i18n (internationalization)

The boilerplate must implement i18n with i18next following a server/client split:

1. **`src/utils/i18n.server.js`** — Server-side i18n initialization without React bindings. `getLocale(request)` resolves locale from URL path segment → cookie → Accept-Language header → default. `getRouteNamespaces(context)` extracts per-route i18n namespaces from route handles. Include RTL direction detection.
2. **`src/utils/i18n.js`** — Client-side i18n initialization with `initReactI18next`. Synchronous init (`initImmediate: false`, `useSuspense: false`). Shared `getDirection(locale)` utility.
3. **`src/configs/i18n.js`** — Supported locales and default locale config.
4. **`src/locales/en.js`** and **`src/locales/es.js`** — Translation bundles as JS objects.
5. **Root route (`root.jsx`)** — Loader calls `getLocale(request)` server-side. Root component sets `<html lang={locale} dir={getDirection(locale)}>`. `useEffect` syncs client i18n language on locale change. Route handles declare `i18n: ['translation']`.

The i18n layer is load-bearing: locale flows from the server loader through the HTML document lang/dir attributes into every component. The scaffold must show this full path working.

## Foundation layer: theme system

The boilerplate must implement a multi-theme system using `data-theme` attribute switching:

1. **`src/components/theme-provider.jsx`** — React context provider. Reads from `localStorage`, sets `document.documentElement.dataset.theme`. Memoized context value. Default theme: `'dark'`. Configurable `themeStorageKey`.
2. **Flash-prevention hydration script** — An inline `<script>` in `<head>` that runs before React hydrates. Reads `localStorage`, sets `data-theme` on `<html>` immediately. This prevents the white flash on pages that default to dark mode.
3. **`src/styles/theme.css`** — CSS custom properties inside `@theme {}` block (Tailwind v4 syntax). Defines color system (background, foreground, card, primary, secondary, accent, muted, border, ring, destructive), responsive typography scale using `clamp()`, font families, spacing tokens, radius tokens.
4. **`src/styles/theme-overrides.css`** — `[data-theme='dark']` and optional additional theme selectors that swap CSS variables.
5. **Root route** wraps `<Outlet>` in `<ThemeProvider>`. HTML element carries `data-theme` attribute.

The theme system is not cosmetic decoration — it is the bridge between DESIGN.md tokens and runtime CSS.

## Foundation layer: modular CSS architecture

The boilerplate must use a barrel-style CSS architecture (not a single monolithic file):

```
src/styles/
├── app.css                          # barrel/index — imports everything in order
├── imports.css                      # @import 'tailwindcss'; @plugin '@tailwindcss/typography'; fonts
├── theme.css                        # @theme {} — CSS custom properties (design tokens)
├── theme-overrides.css              # [data-theme='dark'] { ... } — theme switching
├── base.css                         # @layer base — resets, font smoothing, reduced-motion
├── components.css                   # @layer components — glass cards, scrollbars, custom widgets
├── utilities/
│   ├── layout.css                   # @layer utilities — indent, spacing helpers
│   ├── typography.css               # @layer utilities — type utilities
│   ├── surfaces.css                 # @layer utilities — glassmorphism, surfaces
│   ├── backgrounds-static.css       # @layer utilities — static gradient backgrounds
│   ├── backgrounds-animated.css     # @layer utilities — animated gradient backgrounds
│   └── backgrounds-shaders.css      # @layer utilities — CSS-only shader effects (noise, waves, grid, mesh)
└── keyframes/
    ├── components.css               # component animation keyframes
    ├── backgrounds.css              # background animation keyframes
    └── shaders.css                  # shader animation keyframes
```

Import order matters: imports → theme → overrides → base → components → utilities → keyframes.

The `base.css` MUST include:
- Font smoothing (`-webkit-font-smoothing: antialiased`)
- Text rendering optimization
- OpenType feature settings (kern, liga)
- `prefers-reduced-motion` media query that kills all animations

This architecture scales to hundreds of utility classes without cognitive overload.

## Reference architecture: patterns extracted from production SSR + 3D project

The following patterns come from a real Remix (now React Router v7) project that combined SSR, i18n, theming, p5.js, GLSL shaders, and React Three Fiber. They are proven and should be incorporated into the scaffold.

### DOM / Canvas layer separation (root.jsx)

The root layout must keep DOM UI physically separate from the R3F canvas:

```
<ThemeProvider>
  <div id="dom-ui" className="fixed top-0 left-0 w-full z-10 pointer-events-auto">
    <ModeToggle />
    <LangSelector />
    <NavigationMenu />
  </div>
  <Outlet />  ← routes render Canvas here
</ThemeProvider>
<ScrollRestoration />
```

CSS enforces z-index discipline:
- `#three-canvas` or Canvas wrapper: `position: fixed; z-index: 0;` — fills viewport
- `#dom-ui`: `position: fixed; z-index: 10;` — overlays canvas
- `html, body`: `overflow: hidden; height: 100vh;` — prevents body scroll (scroll is captured by wheel events in the 3D scene)

### ThemeProvider with 3+ themes

The ThemeProvider must support an arbitrary number of themes (not just light/dark). The reference project uses three: `light`, `dark`, `tyrian` (custom branded theme). Implementation:
- `document.documentElement.classList` toggles the active theme class
- CSS custom properties in `@layer base` define tokens per theme (`:root` for default, `.dark` for dark, `.tyrian` for custom)
- `localStorage` persists selection; SSR-safe check (`typeof window !== 'undefined'`)
- ModeToggle component uses a `mounted` state guard to prevent hydration mismatches

### p5.js as live texture pipeline

Scaffold must include a `src/components/canvas/p5/` directory with:

1. **P5Scene.jsx** — wraps a child p5 sketch, creates a `mesh` + `boxGeometry`, and pipes the texture from the child via `cloneElement(children, { onTextureUpdate })`. Uses `useFrame` to sync texture each frame.
2. **P5SketchNoise.jsx** (or similar) — headless p5 sketch using dynamic `import('p5')` in `useEffect`. Creates `createGraphics(w, h, p.WEBGL)`, `pixelDensity(1)`, `frameRate(30)`. Outputs a `THREE.Texture` via `onTextureUpdate` callback. Returns `null`.
3. **SSR guard** — `useHydrated()` or `typeof window` check before p5 initialization.

### CanvasContext for forced-perspective sizing

A React context that computes depth-independent plane dimensions using the perspective camera's FOV:

```
CanvasContext provides: { width, height, scaleFactor, distanceFactor, depth, z }

Computation:
  vFOV = degToRad(camera.fov)
  depth = |camera.position.z - z|
  height = 2 * tan(vFOV / 2) * depth
  width = height * (viewport.width / viewport.height)
  scaleFactor = min(1, height / viewport.height)
  distanceFactor = (1.1 * height) / viewport.height
```

This context wraps HtmlPlane and P5Scene children to ensure every content plane fills the viewport identically regardless of Z position.

### Scroll-driven camera (wheel → depth traversal)

Instead of page scroll, the 3D scene captures `wheel` events and translates cumulative `deltaY` into a `scrollDepth` state (range `[0, -100]`). A `CameraController` component lerps the camera's Z toward the target:

```
useFrame(() => {
  camera.position.z = MathUtils.lerp(camera.position.z, 5 + scrollDepth, 0.1);
});
```

Content planes at z=-1 (foreground), z=-50 (middle), z=-100 (background) create a depth corridor the user travels through.

### SSR pre-rendered HTML in 3D

The reference project pre-renders localized React JSX to HTML string in the server loader using `renderToString`, then injects the result into a Drei `<Html>` component via `dangerouslySetInnerHTML`. This ensures:
- Content is server-rendered, crawlable, and localized
- The 3D scene acts as presentation layer, not content authority
- SEO and a11y are preserved even when the visual experience is 3D

### Custom shader materials

Include `src/materials/` with at least one custom `shaderMaterial` example using `@react-three/drei`'s `shaderMaterial` helper + `extend`:
- Vertex shader with `vUv` passthrough
- Fragment shader sampling a `uTexture` uniform
- Optional `uTime` uniform for animation
- Registered via `extend({ CustomMaterial })` so it's usable as a JSX element

### Hygraph / GraphQL integration point

The reference project uses `graphql-request` + Hygraph for CMS content. The scaffold should include a stub `src/lib/cms/graphqlClient.js` with:
- `GraphQLClient` initialized from an env var endpoint
- Bearer token auth from env var
- A note that this is a real production pattern, not a mock

## Quality requirements

All generated code must:

- be runnable
- use real imports
- avoid placeholder variables unless clearly documented
- avoid fake-complete code
- clean up listeners and Three.js resources
- avoid React state updates inside useFrame hot paths
- include concise comments that explain why, not what

## Performance requirements

Bake the performance philosophy into the scaffold:

- demand frameloop where appropriate
- safe defaults for DPR
- hooks or utilities for renderer.info inspection
- disposal helpers
- no accidental per-frame allocations in obvious hot paths
- no unnecessary shadow casters by default

## Accessibility requirements

Bake accessibility into the scaffold:

- semantic HTML outside the canvas
- reduced-motion detection and fallback
- keyboard-safe structure for route navigation
- no canvas-only critical information
